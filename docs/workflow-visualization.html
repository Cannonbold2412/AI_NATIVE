📹 Recording Phase: Capture

1 Browser Automation & Event Binding
Playwright launches a real Chrome instance with a JavaScript bridge injected via expose_binding. Every user interaction (click, type, navigate) is captured in real-time.
2 Event Structure
Each captured event includes complete context: the action performed, the target element, selectors, semantic meaning, and visual evidence.
{ "action": { "action": "click", "button": "left" }, "target": { "tag": "button", "inner_text": "Submit", "classes": ["btn", "btn-primary"] }, "selectors": { "css": "button.btn-primary", "xpath": "//button[text()='Submit']" }, "visual": { "full_screenshot": "sessions/xxx/1_full.jpg", "bbox": { "x": 100, "y": 200 } }, "page": { "url": "https://example.com", "title": "Login" } }
3 Threading Model
The recorder uses a thread-safe architecture:
  • Driver Thread: Runs Playwright sync API, pumps events
  • Async Thread: Main loop collects and processes payloads
  • Queue: SimpleQueue for thread-safe communication
🔍 Why Screenshots Matter: Each event captures a full-page screenshot and element bounding box. These serve as visual evidence for LLM recovery and human validation.

⚙️ Data Processing Pipeline

1 Validation
Every raw event is validated against the RecordedEvent schema. Missing or malformed fields are caught immediately.
2 Cleaning & Normalization
Text is normalized, whitespace collapsed, selectors canonicalized, and class names sorted alphabetically.
Before
"inner_text": " Submit Form "
"classes": ["btn", "btn-primary", "btn"]
"normalized_text": "Submit Form"
After
"inner_text": "Submit Form"
"classes": ["btn", "btn-primary"]
"normalized_text": "submit form"
3 Deduplication
Consecutive identical scroll events are merged (keeping only the first screenshot). This prevents noise from rapid scroll captures.
4 Semantic Enrichment (via LLM)
For each event, the LLM extracts semantic intent based on element text, type, and page context.
5 Scroll Annotation
Scroll delta is calculated by comparing scroll position before and after each action. This helps identify "wait for scroll" dependencies.
✅ Result: Cleaned, deduplicated, semantically enriched events with confidence scores and pixel-perfect visual evidence.

🧠 LLM Enrichment Layer

1 When LLM Is Called
The LLM is invoked at three critical points in the pipeline:
Phase Purpose Input Output
Pipeline Extract semantic intent Element text, type, page context Intent label, normalized text
Validation Validate intent consistency Action type, target intent Pass/fail + reason
Recovery Fix failed actions Screenshot, error, original selector Alternative selector/action
2 Semantic Enrichment Prompt
The LLM receives a structured input to understand context:
SemanticLLMInput { raw_text: "Submit Login Form", element_type: "button", context: "User authentication page" } LLM Response: { intent: "submit_login_form", normalized_text: "submit login form", confidence: 0.95 }
3 Vision-Based Analysis
For complex UI understanding, the vision_llm analyzes screenshots:
  • Element location detection (coordinates, size)
  • Layout understanding (spatial relationships)
  • Text extraction from images
  • Failure recovery ("The button moved to bottom-right")
⏱️ Timeout Management: Text LLM calls use 2-second timeout (fast). Vision calls use 120-second timeout (large multimodal payloads are slow). If LLM timeout, system falls back to structured selectors.

✏️ Human Editing Interface

1 Editing Workflow
Humans review and refine captured events through the Next.js frontend:
  • Load recorded events from session
  • Display event with full visual preview
  • Modify action, selectors, timing, validation
  • System validates changes against screenshot
  • Save to workflow (event updates persisted)
2 Editable Fields
Each event has multiple aspects that can be refined:
Field Purpose Example
Action Modify type or parameters click → type, button: left → middle
Selectors Switch to more reliable selector CSS → XPath or text-based
Timing Adjust waits for slow pages wait_after: 500ms → 2000ms
Validation Add success criteria URL contains "/dashboard"
3 Validation Rules
Editors can add validation rules to ensure actions succeed:
"validation": { "success_criteria": [ { "type": "url_contains", "value": "/dashboard" }, { "type": "element_visible", "selector": "[data-testid='user-menu']" } ], "timeout_ms": 10000 }
✅ Result: Production-ready event definitions with validated selectors, timing, and success criteria.

🔧 Workflow Compilation

1 Action Extraction
Events are converted to executable Action objects with selectors and parameters:
Action { type: "click" | "type" | "navigate" | "scroll", selector: "button.btn-primary", selectors_fallback: ["button#submit", "//button"], text?: "${email}", wait_before_ms?: 0, wait_after_ms?: 500, selector_confidence: 0.92 }
2 Selector Scoring
Selectors are scored (0.0-1.0) for robustness:
Selector Type Score Impact Example
ID-based CSS +0.2 (most stable) button#submit-btn
Class-based CSS -0.1 (moderate) button.btn.btn-primary
XPath -0.1 (complex) //button[@id='submit']
Text-based -0.2 (fragile) //button[text()='Submit']
3 Intent Validation
Semantic consistency is verified:
✅ Rule 1: Click on button element must have submit/select intent ✅ Rule 2: Type action requires input/textarea element ✅ Rule 3: Navigation action must change URL ✅ Rule 4: All selectors must find exactly 1 element ✅ Rule 5: Form submission must have success validation
4 Dependency Resolution
Actions that depend on prior actions are identified and ordered:
  • Action B depends on A if B's selector includes A's result text
  • Action B depends on A if B's precondition requires A's state_change
  • Topological sort ensures correct execution order
✅ Result: Compiled skill with validation rules, proper sequencing, and fallback selectors.

📦 Skill Package Generation

1 Package Structure
A complete, executable skill package includes:
skill_package/ ├── README.md # Human-readable skill descriptions ├── skills.json # Compiled skill definitions ├── execute.py # Universal Playwright execution engine ├── recovery.py # LLM-based recovery layer ├── vision_recovery.py # Vision-based element detection └── pyproject.toml # Python package metadata
2 README.md Template
Auto-generated documentation describes all available skills:
# Render.com Automation Plugin Complete automation for deploying services on Render.com. ## Available Skills ### 1. Login to Render Authenticate with Render account - **Parameters**: email (string), password (string) - **Success Criteria**: Dashboard visible with user menu ### 2. Create New Service Create service from GitHub repository - **Parameters**: repo_url, service_name, branch - **Success Criteria**: Service appears in dashboard ### 3. Deploy Service Trigger deployment of current code - **Success Criteria**: Deployment logs visible, status = "live"
3 Execution Engine (execute.py)
Universal Python module handles skill execution:
  • Load skill definition from skills.json
  • Launch Playwright browser
  • Execute each step sequentially
  • Validate after each step
  • On failure, trigger 4-layer tiered recovery (L1→L2→L3→L4)
  • Layer 1-3: Instant (selectors, anchors, text variants)
  • Layer 4: Vision LLM (120s timeout, for major UI changes)
📦 Distribution: The skill_package can be published to GitHub with version control, discovered by other users, and executed via Claude agent or API.

🛡️ Self-Healing 4-Layer Tiered Recovery System

⚡ Overview: When action execution fails, Conxa tries 4 recovery layers in sequence. Layers 1-3 are instant (CSS/DOM queries), Layer 4 uses vision AI (120s timeout). Combined success rate: ~95%.
L1 Selector Context (Instant)
Try primary and alternative CSS selectors:
{ primary: "button#submit", alternatives: [ "button.btn-primary", "button[type='submit']", "[aria-label='Submit']" ] } Success Rate: 60-70% (simple selector changes) Time: Instant (synchronous DOM query)
L2 Text Anchors (Instant)
Search using recorded text anchors with priority scoring:
{ anchors: [ { text: "Submit", priority: 1.0 }, { text: "Send", priority: 0.8 }, { text: "Confirm", priority: 0.6 } ] } Success Rate: 50-60% (minor text changes) Time: Instant (DOM text search)
L3 Text Fallback Variants (Instant)
Try normalized text variations and element role:
{ fallback: { text_variants: [ "submit", "send", "confirm" ], role: "button" } } Success Rate: 40-50% (text normalization) Time: Instant (normalized search)
L4 Vision Recovery (⏱️ 120 seconds)
Use vision LLM to analyze screenshot and locate element:
{ visual_ref: "Image_3.jpg", visual_metadata: { step_id: 3, source: "stored_step_visual", available: true } } Process: 1. Analyze current screenshot with vision LLM 2. Detect all interactive elements on page 3. Score candidates against original recorded intent 4. Return best-matching element's selector Success Rate: 75-85% (major UI changes) Time: ~120 seconds (vision model inference)
Complete Recovery Flow
Sequential fallback for a failed action:

📊 Recovery Statistics:
  • Layers 1-3 combined: ~80% of failures (instant resolution)
  • Layer 4 (vision): Handles remaining 20% (complex UI changes)
  • Combined success: ~95% of all failures
  • Retry policy: 2 attempts for element actions, 3 for navigate/check
