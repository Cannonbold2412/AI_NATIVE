Input Contract

The Python function accepts a JSON string, not a Python object. The frontend parses and validates text first, then re-serializes it into the API body field json_text. 

build_skill_package(
  json_text: str,
  package_name: str | None = None,
  *,
  bundle_slug: str | None = None,
  realtime_sink: Callable[[dict[str, Any]], None] | None = None,
) -> dict[str, Any]
InputMeaningUsed for
json_text Raw workflow/package JSON as text. Parsed, cleaned, workflow-enumerated, then sent to the structuring LLM as sanitized raw steps.
package_name Optional workflow folder name hint. Only applied directly when the payload contains exactly one workflow. Multi-workflow payloads derive names from workflow titles.
bundle_slug Optional bundle folder under output/skill_package/. Defaults through slug normalization to default.
realtime_sink Optional callback for build-log entries. Used by SSE routes to stream progress such as LLM request, response, retry, and phase events.

Accepted JSON Shapes

The extractor accepts a direct list of step objects, a root object with common step keys, or containers such as skills, workflows, flows, scenarios, and recordings. 

{
  "title": "Delete database",
  "steps": [
    { "type": "navigate", "url": "https://example.test/login" },
    { "type": "fill", "selector": "input[name=email]", "value": "me@example.test" },
    { "type": "click", "selector": "text=Sign in" }
  ]
}
{
  "workflows": [
    { "title": "Login", "steps": [ ... ] },
    { "title": "Delete database", "steps": [ ... ] }
  ]
}

End-to-End Flow

Start log scope.

build_skill_package opens skill_pack_build_log_scope. All phase rows and LLM rows are collected into build_log. If realtime_sink is supplied, each row is also pushed to the sink.

Parse JSON.

_parse_json_text calls json.loads. The root must be an object or array. Invalid JSON raises ValueError, later wrapped as SkillPackBuildUserError.

Preprocess package JSON.

For dictionary payloads, _prepare_skill_package_payload captures title/session hints, calls preprocess_plugin_json, restores missing useful title/session metadata, then removes declaration blocks such as inputs, parameters, params, and variables throughout the tree.

Resolve bundle slug.

_resolve_bundle_slug slugifies the requested bundle name and rejects reserved or malformed names.

Enumerate workflow recordings.

_enumerate_raw_workflows returns one RawWorkflow per detected workflow. Each item carries title, the workflow payload, and extracted raw step objects.

Compile each workflow sequentially.

_compile_skill_package_payloads loops with a normal Python for. For each workflow it calls _compile_workflow_payload. There is no asyncio.gather, worker pool, batching, or parallel LLM fanout in this path.

LLM structure raw steps.

structure_steps_with_llm strips screenshots and heavy visual fields from the raw steps, sends them to _call_structuring_llm, then validates the returned goal and canonical step list.

Deterministic compile.

After the LLM returns structured steps, the builder deterministically derives inputs, execution plan, manifest, and SKILL.md. It validates that execution includes at least one fill and one click, uses allowed step types, avoids XPath, and has valid selectors.

Collect visual assets.

_collect_visual_assets reads screenshot paths from the original cleaned payload and session events, then copies available image bytes into names like Image_0.png and Image_3.jpg. These images are not sent to the structuring LLM.

Generate recovery map deterministically.

generate_recovery compiles the structured steps again, builds one recovery entry for each recovery-eligible action, attaches matching visual_ref values when images exist, and validates anchors, fallbacks, and selectors.

Serialize workflow artifacts.

The builder serializes input.json, manifest.json, execution.json, and recovery.json with indentation. SKILL.md and tests/test-cases.json are assembled separately.

Persist workflow folders.

_persist_compiled_workflows writes each compiled workflow under output/skill_package/<bundle>/workflows/<workflow>/, writes visual assets, then refreshes bundle index/runtime files.

Return API-facing result.

_format_build_skill_package_result returns the first workflow summary plus the full list of workflow names, refreshed bundle index JSON, bundle entry markdown, and the full build_log.

LLM Calls

The only LLM call made directly by build_skill_package

The build path calls the pack-specific structuring LLM through _call_structuring_llm. This call converts noisy browser logs into a compact structured workflow.

ItemExact behavior
Trigger Once per detected workflow, inside _compile_workflow_payload.
Endpoint settings.pack_llm_endpoint, normalized to /v1/chat/completions when needed.
Model settings.pack_llm_model.
HTTP method Synchronous urllib.request.urlopen POST.
Auth Authorization: Bearer <key> if SKILL_PACK_LLM_API_KEY or a rotated key from SKILL_PACK_LLM_API_KEYS exists.
Timeout settings.pack_llm_timeout_ms, default currently 360000 ms.
Retries At most 3 HTTP POST attempts total. Retries only for HTTP 502, 503, and 504.
JSON mode Adds {"type":"json_object"} as response_format except for hosts containing integrate.api.nvidia.com.
Prompt System prompt instructs the model to remove focus/wait noise, infer intent, replace hardcoded values with variables, and output only allowed step types.

LLM Request Shape

The request body is OpenAI-compatible chat-completions JSON. The user message is a minified JSON string containing only sanitized raw steps. 

{
  "model": "<settings.pack_llm_model>",
  "messages": [
    { "role": "system", "content": "<structuring system prompt>" },
    {
      "role": "user",
      "content": "{\"raw_steps\":[ ... sanitized step objects ... ]}"
    }
  ],
  "temperature": 0.0,
  "max_tokens": "<optional settings.pack_llm_structure_max_tokens>",
  "top_p": "<optional settings.pack_llm_top_p>",
  "response_format": { "type": "json_object" }
}

What Is Removed Before Sending to the LLM

sanitize_raw_steps_for_llm deep-copies each step and removes heavy or session-specific fields:

  • screenshot
  • visual
  • signals.visual
  • extras.session_id

Expected LLM Output

The builder accepts a top-level object with goal and steps. If the provider response is a normal chat-completions object, it extracts choices[0].message.content and parses the JSON object from there.

{
  "goal": "Sign in and delete a database",
  "steps": [
    { "type": "navigate", "url": "https://example.test/login" },
    { "type": "fill", "selector": "input[name=email]", "value": "{{user_email}}" },
    { "type": "fill", "selector": "input[name=password]", "value": "{{user_password}}" },
    { "type": "click", "selector": "text=Sign in" },
    { "type": "check", "kind": "url", "pattern": "/dashboard" },
    { "type": "click", "selector": "text=Delete" },
    { "type": "check", "kind": "text", "text": "Deleted" }
  ]
}

Allowed Structured Step Types

The validator accepts navigate, fill, type, click, select, focus, scroll, and check. The prompt tells the LLM not to output wait. The final execution validator also rejects wait steps.

Important: app.llm.anchor_vision_llm and app.llm.client.call_llm support multimodal anchor generation and optional parallel fanout for anchor_vision, but that path is not called by build_skill_package. In this builder, recovery entries and anchors are generated deterministically in generate_recovery.

Sync, Async, Parallel, Batched

QuestionAnswer for build_skill_packageWhere it happens
Is the core builder synchronous? Yes The function and its internal LLM HTTP call are normal blocking Python code. build_skill_package, _call_structuring_llm
Is the API streaming route async? Yes The streaming route is async, but it runs the blocking builder in an executor thread and reads a queue with asyncio.to_thread(q.get). _skill_pack_build_sse_events
Are multiple workflows compiled in parallel? No Multi-workflow payloads are compiled in a sequential for loop. _compile_skill_package_payloads
Are LLM structuring calls batched? No Each workflow sends its own {"raw_steps":[...]} request. There is no combined batch API call. _compile_workflow_payload to structure_steps_with_llm
Are LLM structuring calls parallel? No The pack structuring LLM uses one blocking POST per attempt. _call_structuring_llm
Does API-key rotation mean parallelism? No Each attempt selects one pack key via round-robin. It does not fan out to all pack keys. next_pack_api_key()
Does the frontend fetch selected source workflows in parallel? Yes, before build The UI can fetch multiple existing workflows with Promise.all, then it builds/appends workflows one request at a time. SkillPackBuilderPage.tsx
Does generic LLM client have parallel fanout? Only outside this path app.llm.client can parallelize anchor_vision across keys, but build_skill_package does not use that call. app/llm/client.py

Validation and Errors

Most validation failures are raised as ValueError inside the transaction and wrapped by SkillPackBuildUserError, preserving any accumulated build_log. HTTP routes convert this into a 400 response with detail.message and detail.build_log.

FailureCondition
Invalid JSONjson_text cannot be parsed or root is not object/array.
No stepsNo workflow steps are found after extraction.
LLM disabled/missing configSKILL_PACK_LLM_ENABLED is false, endpoint is empty, or model is empty.
LLM transport failureNetwork, timeout, or non-retryable HTTP failure. HTTP 502/503/504 retry up to 3 attempts total.
Bad LLM JSONProvider response cannot be parsed as JSON or chat content does not contain a JSON object.
Bad structured outputMissing goal, missing steps, unsupported type, invalid selector, missing required fields.
Bad execution planNo steps, missing at least one fill or one click, wait step, XPath, generic selectors, invalid navigate/check/scroll values.
Bad recovery mapMissing recovery entry, empty anchors, generic labels, invalid selector alternatives, unexpected validation/scroll data in recovery entries.
Name collision on appendappend_workflow_to_skill_package rejects workflow names already present in the target bundle.

Build Log Events You Should Expect

  • bundle_compile_outline: number of workflows found.
  • workflow_compile_start and workflow_compile_complete: per workflow lifecycle.
  • pipeline_phase: phase start/done for llm_structure, deterministic_compile, recovery_map, and serialize_artifacts.
  • llm_request_sent: attempt number, model, host, path, timeout, payload size, raw step count, max attempts, JSON-mode flag.
  • llm_response_received: elapsed milliseconds and response character count.
  • llm_http_error, llm_retry, llm_timeout, llm_network_error: failure and retry diagnostics.
  • llm_response_parsed: parsed provider response metadata.
  • persist_phase: persistence start/done and workflow names.

API and Frontend Path

Non-streaming build

Frontend buildSkillPackage(...)
  - parseJsonSource(jsonText)
  - validateSource(payload)
  - POST /skill-pack/build
Backend post_build_skill_pack(...)
  - build_skill_package(body.json_text, package_name=body.package_name, bundle_slug=body.bundle_name)
  - return result dict

Streaming build

Frontend buildSkillPackage(..., { onLog })
  - POST /skill-pack/build/stream
Backend post_build_skill_pack_stream(...)
  - starts async StreamingResponse
  - run_in_executor(None, runner)
  - runner calls blocking build_skill_package(..., realtime_sink=realtime_sink)
  - realtime_sink puts log rows into SimpleQueue
  - async generator emits SSE:
      data: {"event":"log","entry":{...}}
      data: {"event":"done","result":{...}}
      data: {"event":"error","message":"...","build_log":[...]}
The streaming route gives live progress, but it does not make the internal compile itself async or parallel. It moves the blocking build onto an executor thread so the HTTP connection can stream queue events.
