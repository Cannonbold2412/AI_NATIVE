📊 Data Flow & Transformations

How raw events evolve through the Conxa pipeline

1️⃣ Recording: Raw Event Capture
📹 Raw Event (Just Captured)

When a user clicks a button, the JavaScript bridge captures this raw data:

{ "action": { "action": "click", "button": "left", "count": 1 }, "target": { "tag": "button", "inner_text": " Submit Form ", "classes": ["btn", "btn-primary", "btn"], "attributes": { "id": "submit-btn", "type": "submit" } }, "selectors": { "css": "button.btn.btn-primary", "xpath": "//button[@id='submit-btn']", "text_xpath": "//button[contains(text(), 'Submit')]" }, "semantic": { "normalized_text": "Submit Form", "input_type": null }, "visual": { "full_screenshot": "sessions/xxx/1_full.jpg", "bbox": { "x": 100, "y": 200, "w": 80, "h": 40 } } }
📌 Raw Event Size: ~50-100 KB per event (large due to full screenshots)
2️⃣ Pipeline Processing: Cleaning
⚙️ Transformation 1: Text Normalization & Deduplication
Changes Applied:
Collapse whitespace in text fields
Deduplicate CSS classes
Sort classes alphabetically
Normalize text to lowercase for semantic matching
❌ BEFORE
inner_text: " Submit Form "
classes: ["btn", "btn-primary", "btn"]
normalized: "Submit Form"
✅ AFTER
inner_text: "Submit Form"
classes: ["btn", "btn-primary"]
normalized: "submit form"
⚙️ Transformation 2: Selector Canonicalization
Changes Applied:
Validate CSS selector syntax
Generate simplest CSS version
Verify XPath correctness
Compute selector specificity scores
❌ BEFORE
css: "button.btn.btn-primary"
xpath: "//button[@id='submit']"
text: "//button[contains()]"
✅ AFTER
css: "button#submit-btn"
xpath: "//button[@id='submit']"
text_xpath: "//*[text()='Submit']"

4️⃣ Human Editing: Refinement
✏️ Transformation 4: Validation & Parameter Addition
Human Edits:
Verify LLM intent matches actual action
Adjust selectors for better stability
Add validation rules (success criteria)
Fine-tune timing (wait_before, wait_after)
❌ AUTO-GENERATED
action: "click"
selector: "button.btn"
wait_after: 500ms
validation: null
✅ HUMAN-EDITED
action: "click"
selector: "button#submit"
wait_after: 2000ms
validation: {
  url_contains: "/dashboard"
}
💡 Why Editing Matters: Selectors based on IDs are more stable than class-based. Longer wait times accommodate slow networks. Validation rules catch failed actions early.
5️⃣ Compilation: Skill Generation
🔧 Transformation 5: Action Extraction & Selector Ranking
Compiler Processes:
Convert event to executable Action type
Score selectors by stability (0.0-1.0)
Rank best selector first, add fallbacks
Embed validation rules in step definition
Selector Type Score Ranking Fallback?
button#submit 0.95 1st (Best) Primary
button.btn-primary 0.75 2nd Fallback 1
//button[@type='submit'] 0.68 3rd Fallback 2
//button[text()='Submit'] 0.45 4th (Worst) Last Resort
"compiled_action": { "id": "step_1", "action": "click", "selector": "button#submit", "selector_confidence": 0.95, "selectors_fallback": [ "button.btn-primary", "//button[@type='submit']" ], "validation": { "success_criteria": [ { "type": "url_contains", "value": "/dashboard" } ] }, "wait_after_ms": 2000 }
6️⃣ Execution: Runtime Transformation
⚡ Transformation 6: Action Execution & Validation
At Runtime:
Load skill from skills.json into memory
For each action, find element using best selector
If element not found, try fallbacks
Execute action (click, type, navigate, etc.)
Wait and check success criteria
❌ EXECUTION ATTEMPT 1
Selector: "button#submit"
Result: NOT FOUND
Status: Failed
→ Try Fallback 1
✅ EXECUTION ATTEMPT 2
Selector: "button.btn-primary"
Result: FOUND (1 match)
Action: Click
Status: Success ✓
Self-Healing in Action: When the primary selector fails, the system automatically tries fallback selectors without requiring manual intervention.
